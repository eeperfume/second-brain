# 사용자 상태 변경과 토큰 즉시 무효화

[[session-and-jwt]]에서 차단할 access token의 ID를 거부 목록(denylist)에 올려 요청마다 대조하는 선택지를 봤고, [[token-auth-server-spring]]에서 jti를 남은 수명만큼 Redis에 올리는 코드로 구현했다. 그렇다면 사용자 상태 변경(휴면, 잠금, 차단)에 따른 즉시 토큰 무효화도 denylist가 유일한 해결책인가?

## 거부 목록은 토큰 중심, 사용자 상태 변경은 사용자 중심이다

denylist는 토큰 jti를 하나씩 지목하는 토큰 중심 장치다. 그런데 사용자 상태 변경은 사용자 중심 사건이다. 사용자를 차단하면 그 사용자의 모든 토큰이 차단되어야 하는데, 서버는 무상태로 발급한 access token을 애초에 추적하지 않으니 개별 토큰을 열거할 방법도 없고 열거할 필요도 없다. 사용자별 확인이나 사용자 수준의 거부 목록으로 충분할 것이다.

반대로 사건이 토큰 하나에 그치는 경우(이 기기만 로그아웃, 노출이 토큰 하나로 확인된 탈취)는 여전히 jti denylist가 맞다. 사용자 단위 무효화는 다른 기기의 멀쩡한 세션까지 함께 끊어 버리기 때문이다. 이 노트는 사건이 사용자 단위일 때의 이야기다.

## 사용자 중심 무효화 선택지

**1. 사용자 상태 조회 (per-request check)**

매 요청마다 `user:{id}:status`를 Redis에서 조회한다. 휴면/잠금/차단 처리 시 그 키만 갱신하면 즉시 반영된다. 토큰을 하나하나 추적할 필요가 없어서 denylist보다 오히려 단순할 수 있다. 다만, 조회 비용은 jti denylist 조회와 사실상 동일하다(둘 다 요청당 Redis 조회 1회가 발생한다).

**2. 사용자 단위의 denylist**

denylist를 쓰더라도 토큰 jti가 아니라 user_id를 키로 넣는 변형이다. "이 사용자의 모든 토큰을 무효화"하겠다는 시멘틱이라 상태를 변경하는 시나리오에는 더 적합하다. Redis에 `denied:user:{id}` + access token 최대 수명만큼(시계 오차 여유를 조금 더해) TTL을 걸면 된다. 차단 시점 이전에 발급된 토큰은 그 시간이 지나면 전부 스스로 만료되기 때문이다.

다만, 여기엔 함정이 하나 있다. `denied:user:{id}`를 jti denylist처럼 존재 여부만 보는 키로 만들면(값은 [[token-auth-server-spring]]의 `"1"`처럼 의미 없는 것을 넣고, 검사는 "키가 있나 없나"만 보는 방식) 차단 "해제"를 다루지 못한다.

타임라인으로 보면 분명해진다. access token 수명이 15분이라고 하자.

- 09:55 사용자가 access token을 발급받는다(10:10까지 유효).
- 10:00 이 사용자를 차단한다. `denied:user:{id}` 키를 올린다.
- 10:05 사정이 바뀌어 차단을 해제하기로 한다.

이때 키를 지우면 09:55에 나간 토큰(10:10까지 유효)이 다시 통과된다. 차단 이유가 토큰 탈취였다면 공격자가 쥔 그 토큰이 되살아나는 것이다. 반대로 키를 안 지우면, 해제 후 새로 로그인해서 정상 발급받은 토큰까지 TTL이 끝날 때까지 거절된다. "차단 전 토큰은 죽이고, 해제 후 토큰은 살린다"는 구분이 필요한데, 키가 있다/없다만으로는 그 구분을 표현할 수 없기 때문이다.

그래서 값에 차단 시각을 저장해 두고 토큰의 발급 시각(iat)과 비교하는 변형이 낫다. 규칙이 "이 사용자를 막아라"가 아니라 "이 시각 이전에 발급된 토큰은 전부 무효"가 되는 것이다. 위 타임라인이라면 09:55 발급 토큰은 iat가 차단 시각(10:00)보다 앞이라 거절되고, 해제 후 새로 발급받은 토큰은 iat가 그보다 뒤라 통과된다. 키를 지울 필요도 없고, 재차단할 땐 값을 새 차단 시각으로 갱신하면 된다(Firebase의 tokensValidAfterTime이 이 방식이다).

**3. 토큰 버전 방식**

사용자마다 token_version을 저장하고 토큰 발급 시 클레임(Claim)에 넣는다. 상태 변경 시 버전을 올리면 기존 토큰은 전부 무효화된다. 이것도 매 요청마다 Redis 조회 1회가 발생한다는 점은 같지만, "차단"과 "전체 로그아웃"과 "비밀번호 변경"을 하나의 메커니즘으로 통합할 수 있다(ASP.NET Identity의 SecurityStamp가 같은 패턴이다). 다만, 버전은 계속 사는 데이터라 유실되지 않게 저장해야 하고(유실 후 기본값으로 돌아가면 폐기했던 토큰이 되살아난다), 기기 하나만 골라 로그아웃하는 데는 쓸 수 없다. 버전을 올리면 전 기기가 함께 로그아웃된다.

**4. opaque 토큰 + 서버 세션 (allowlist)**

애초에 토큰 방식이 아니라 세션을 서버가 들고 있으면, 사용자 상태 변경 시 세션을 지우거나 세션 검증 시 사용자 상태를 함께 보면 된다. 이 경우 사용자를 즉시 차단할 수 있다. 다만 이것은 [[session-and-jwt]]의 갈림길에서 세션 쪽 끝을 다시 고르는 것이다.

**공통 유의점 두 가지.**

첫 번째로 조회 횟수가 같다고 성질까지 같은 건 아니다. denylist는 차단된 사용자만 담고 "키가 없으면 통과"가 규칙이라, 장애로 저장소 데이터가 날아가면 차단했던 사용자가 전부 다시 통과된다. 잘못될 때 막아야 할 쪽을 열어 주는 방향으로 잘못되는 것이다(이런 성질을 fail-open이라 부른다). 반대로 상태 조회와 토큰 버전은 전 사용자의 키가 저장소에 늘 있어야 하고, 키가 없을 때 어떻게 할지(원본 DB로 넘어갈지, 거절할지)를 따로 정해야 한다.

두 번째로 어느 방식을 쓰든 신규 발급 경로가 함께 막혀야 한다. 로그인이야 당연히 차단 계정을 거절하겠지만, refresh 경로가 사용자 상태를 확인하지 않으면 denylist 항목이 사라진 뒤에도 오래 사는 refresh token으로 새 access token이 계속 나온다. 더 표준적인 결합은 차단 시점에 그 사용자의 refresh token family를 즉시 회수하는 것이고([[refresh-token-rotation]]), 발급 경로의 상태 확인은 그 위에 겹치는 이중 방어다. 다만 [[token-auth-server-spring]]의 구현에는 refresh token 키와 family 키만 있어서, 사용자 ID로 "이 사용자의 family 전부"를 찾을 방법이 없다. 차단 시점에 회수하려면 사용자별 family 목록(예: `user:{id}:families` Set)을 발급할 때 같이 저장해 둬야 한다.

## 즉시를 엄격하게 준수하면 무상태는 논리적으로 불가능하다

결국 "즉시 사용자를 차단한다"는 요구를 엄격하게 준수하는 순간, 모든 요청이 어떤 형태로든 최신 상태를 참조해야 하니 순수한 무상태(stateless) 검증은 논리적으로 불가능해진다. 왜냐하면 서명만 보고 통과시키는 토큰 방식은 발급 시점의 스냅샷이고, 발급 이후의 상태 변경을 반영할 방법이 자체적으로 없기 때문이다. [[session-store]]에서 완전한 무상태와 즉시 무효화를 동시에 100% 가질 수는 없다고 한 그 트레이드오프다.

정확히 말하면 무상태가 못 하는 것은 "발급 시점에 예측할 수 없었던 변경"의 즉시 반영이다. 휴면 전환처럼 예정 시각을 발급 때 이미 알 수 있는 변경은 만료(exp)에 미리 반영할 수 있으니 이 한계에 걸리지 않는다. 그리고 요청마다 확인하는 장치조차 닿지 않는 구멍이 하나 있는데, WebSocket 같은 장기 연결이다. 접속을 맺을 때 한 번 검증하고 계속 쓰니, 차단할 때 연결을 강제로 끊는 처리가 따로 필요하다.

다만 "포기"의 정도에는 스펙트럼이 있다.

## 포기의 스펙트럼

첫 번째로 네트워크 조회 없이 상태를 참조하는 방법이 있다. denylist나 사용자 상태를 각 서버의 메모리에 복제해 두고, 상태 변경 시 Redis의 pub/sub이나 이벤트 스트림으로 전파하는 방식이다. 검증 자체는 여전히 로컬에서 끝나니 요청당 네트워크 I/O가 없고, 전파 지연은 같은 데이터센터 안이라면 보통 밀리초 수준(보수적으로 잡아도 수십 ms 안)이라 실용적인 측면에서 봤을 때 "즉시"로 취급할 수 있다. 메모리 부담도 작다. 차단 항목은 access token 최대 수명(예: 15분)이 지나면 지워도 되는데, 차단 전에 발급된 토큰은 그 안에 어차피 전부 만료되기 때문이다. 그래서 로컬 메모리에 남는 것은 최근 15분 사이에 발생한 차단 건수뿐이고, 누적 차단자 수가 아무리 많아도 부담이 그 이상 커지지 않는다. 즉 stateless가 주는 이점은 포기하지만 "조회 비용이 없다/네트워크가 발생하지 않는다"는 실질적인 이점은 상당 부분 지킬 수 있다.

두 번째로 실무에서 더 흔한 선택은 "즉시"를 완화하는 쪽이다. access token TTL을 5분 정도로 짧게 잡으면 차단 반영이 최대 5분 지연되는 걸 감수하는 대신 stateless를 온전히 유지하게 된다. access token을 재발급 받는 refresh 시점에는 어차피 서버 조회가 일어나니 그때 상태를 확인하면 된다. 최악은 refresh 직후에 차단되는 경우로, 방금 나간 access token의 수명만큼 늦는다. 실제로 많은 시스템이 "일반 상태 변경은 TTL 만료로 충분하고, 보안 사고급 차단만 즉시 장치"라는 이원화를 택한다. 경계는 사건의 이름이 아니라 "그 지연 동안 최악에 무슨 일이 벌어질 수 있는가"로 긋는다. 권한 강등이 일반 변경처럼 보여도, 강등된 관리자가 5분간 관리자 권한으로 움직이는 것이 사고인 시스템이라면 즉시 쪽이다.

정리하면 트레이드오프 자체는 피할 수 없지만, 두 가지 중에 선택할 수 있다. "즉시 차단"을 엄격하게 지키면서 조회 비용을 낮추거나(로컬 복제), "조회 없음"을 지키면서 즉시성을 바운디드 딜레이(Bounded delay, 제한된 지연)로 완화하거나.

- 바운디드 딜레이(Bounded delay, 제한된 지연)는 지연 시간의 최대값이 미리 정해진 한계(여기서는 access token 수명)를 초과하지 않도록 보장되는 상태를 뜻한다.

결국 어느 쪽을 택할지는 "차단이 몇 분 혹은 몇 초 늦어지면 실제로 무슨 일이 생기는가"라는 요구사항의 강도에 달려 있다.

## 첨언: pub/sub에 따라오는 디테일

Redis의 pub/sub을 사용할 때 따라오는 디테일이 하나 있다. 바로 TTL이다. Redis 키에는 TTL을 걸 수 있지만 각 서버의 메모리(예: Set)는 자동으로 안 사라진다는 문제가 있다. 그래서 로컬에도 만료 시각을 같이 저장해서 주기적으로 청소하거나, 항목마다 만료 시각을 들고 조회 시 확인하는 식으로 처리해야 한다. 실무에서는 이 만료 로직을 내장한 Caffeine 같은 로컬 캐시 라이브러리를 쓰는 것이 보통이고, 그 내부도 결국 이 두 방식이다.

그리고 트레이드오프도 하나 생긴다. pub/sub은 전달 보장이 없다(at-most-once). 그래서 네트워크 순단으로 메시지를 놓친 서버는 차단 사실을 모르는 채로 계속 통과시킬 수 있다. 그래서 실무에서는 pub/sub push에 더해 "몇 초 ~ 몇 분마다 Redis 원본과 로컬을 재동기화"하는 폴링을 안전장치로 둔다. 즉 push로 즉시성을 얻고, 주기적 pull로 정합성을 보정하는 이중 구조다.
