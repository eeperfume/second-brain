# JWKS: 공개키를 나눠 주는 창구

[[jwt-signing-and-keys]]에서 RS256(비대칭)은 개인키로 서명하고 공개키로 검증한다고 했다. 그러면 그 공개키를 검증자들에게 어떻게 나눠 줄 것인가. 이 문제를 푸는 표준이 JWKS다.

## 도장 비유

발급 서버가 도장을 찍어 토큰을 낸다고 하자([[session-and-jwt]]에서 서명을 위조 방지 도장에 빗댔다). 검증자는 그 도장이 진짜인지 봐야 하는데, 그러려면 "정품 도장의 무늬"를 알아야 한다. 발급 서버는 이 무늬(공개키)를 숨길 필요가 없다. 그래서 누구나 볼 수 있는 게시판에 붙여 둔다. 검증자는 그 무늬를 한 번 베껴 와서, 이후 들어오는 도장을 자기가 직접 대조한다. 이 게시판이 JWKS다.

## JWKS는 공개키를 담은 JSON 하나

그 게시판을 실제로 열어 보면 JSON 하나다. JWKS(JSON Web Key Set)는 이름 그대로 공개키들을 담은 JSON이다. 발급 서버가 이 JSON을 URL 하나로 공개한다. 주소는 관례상 `/.well-known/jwks.json`이다. 그 주소를 열면 이렇게 생겼다.

```json
{
  "keys": [
    { "kid": "2024-05", "kty": "RSA", "alg": "RS256", "use": "sig",
      "n": "0vx7... (공개키 값)", "e": "AQAB" }
  ]
}
```

- `keys`: 공개키 배열. 여러 개일 수 있다(뒤의 키 교체에서 이유가 나온다). 항목 하나가 공개키 하나다.
- `kid`(key id): 이 공개키에 붙은 이름표. 여러 키 중 어느 것인지 가리킨다.
- `kty`(key type): 키 종류. JWKS에선 보통 `RSA`나 `EC`다. 알고리즘을 뜻하는 `alg`와는 다른 필드다.
- `alg`: 이 키로 하는 서명 방식. `RS256`이면 RSA와 SHA-256을 쓴다. 방식 자체는 [[jwt-signing-and-keys]]에서 다룬다.
- `use`: 이 키의 용도. `sig`는 서명 검증용, `enc`는 암호화용이다. JWT 검증은 서명이라 `sig`.
- `n`, `e`: RSA 공개키의 실제 값. 이 둘이 따로가 아니라 합쳐서 공개키 하나를 이룬다(모듈러스 `n`, 공개 지수 `e`). 좌표 `(x, y)`가 점 두 개가 아니라 점 하나이듯이다.

숨길 값이 아니므로 이 JSON은 공개된 URL로 아무나 받아 갈 수 있다.

## 검증자는 JWKS를 어떻게 쓰나

발급 서버가 공개키를 JWKS에 올리고, 검증 서버가 그것을 받아 이후 스스로 토큰을 검증한다. 아래 그림이 그 전체를 보여준다.

![[jwt-jwks-flow.svg]]

흐름은 세 단계다.

1. 검증 서버는 시작할 때(또는 주기적으로) 그 JWKS URL을 한 번 GET해서 공개키를 자기 프로세스 메모리에 캐시한다. 여기서 메모리는 그 서버 자신의 RAM이고 Redis 같은 별도 저장소가 아니다. 매 요청마다 받지 않는다.
2. 토큰이 오면 토큰 헤더의 `kid`를 본다. 캐시에서 같은 `kid`의 공개키를 고른다.
3. 그 공개키로 서명을 검증한다. 인증 서버를 부르지 않고 검증 서버가 로컬에서 끝낸다.

즉 검증 서버는 공개키만 한 번 받아 두면, 그다음부터는 남에게 묻지 않고 스스로 토큰을 검증한다.

## kid와 키 교체

공개키를 하나만 두는 게 아니라 여러 개를 동시에 둘 수 있고, `kid`로 구분한다. 이것이 키 교체를 매끄럽게 한다.

예를 들어 키를 교체하는 중이면 JWKS에 옛 키와 새 키가 함께 들어 있다. `2024-05`가 옛 키, `2025-01`이 새 키다.

```json
{
  "keys": [
    { "kid": "2024-05", "kty": "RSA", "alg": "RS256", "use": "sig",
      "n": "0vx7... (옛 공개키)", "e": "AQAB" },
    { "kid": "2025-01", "kty": "RSA", "alg": "RS256", "use": "sig",
      "n": "3fJ9... (새 공개키)", "e": "AQAB" }
  ]
}
```

발급자가 새 키(새 `kid`)를 JWKS에 추가하고, 새로 발급하는 토큰부터 새 키로 서명한다. 검증자는 처음 보는 `kid`를 만나면 JWKS를 다시 받아 새 공개키를 얻는다. 옛 공개키는 옛 토큰이 만료될 때까지 JWKS에 남겨 둔다. 그러면 옛 토큰도 새 토큰도 모두 검증된다. HS256에서 비밀키를 모든 서버에 동시에 다시 뿌려야 했던 것과 대비된다. 여기서는 JWKS 한 곳만 갱신하면 검증자들이 알아서 따라온다.

## 코드로 보기

직접 짤 일은 거의 없다. `jose` 같은 라이브러리가 URL에서 공개키를 받아 캐시하고, `kid`를 맞춰 검증까지 해준다.

```js
import { createRemoteJWKSet, jwtVerify } from 'jose'

const JWKS = createRemoteJWKSet(new URL('https://인증서버/.well-known/jwks.json'))
const { payload } = await jwtVerify(token, JWKS, { algorithms: ['RS256'] })
```

Auth0·Keycloak·Cognito 같은 인증 제공자는 이 JWKS URL을 이미 제공한다. OIDC를 쓰면 `/.well-known/openid-configuration`이 그 `jwks_uri` 주소를 알려준다.

## 요약

RS256에서 공개키를 검증자들에게 나눠 주는 표준 방법이 JWKS다. 발급자가 공개키들을 `/.well-known/jwks.json`에 JSON으로 공개하고, 각 키에 `kid` 이름표를 붙인다. 검증자는 이 JSON을 한 번 받아 캐시하고, 토큰의 `kid`로 맞는 공개키를 골라 로컬에서 검증한다. 키 교체도 JWKS 한 곳만 갱신하면 되므로, 비밀키를 모두에게 복제해야 하는 HS256의 부담이 사라진다.
