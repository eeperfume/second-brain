# 세션과 JWT

인증이라는 문제 하나에서 세션과 JWT가 어떻게 갈라지는지를 순서대로 따라간다. 세션 쪽 이야기는 [[cookie-and-session]]과 [[session-store]]에서 다루므로, 여기서는 그 갈림에서 JWT로 뻗는 쪽에 집중한다.

## HTTP는 상태가 없다

모든 이야기가 여기서 시작한다. 웹의 기본 통신 규약인 HTTP는 요청 하나가 끝나면 그 사실을 잊는다. 방금 로그인한 요청과 1초 뒤 들어온 요청이 같은 사람인지 서버는 알 방법이 없다. 요청마다 처음 보는 상대다.

그런데 서비스는 요청마다 "이 요청이 로그인한 그 사용자다"를 알아야 한다. 그러려면 요청마다 아까 그 사용자임을 증명할 무언가를 함께 실어 보내야 한다. 인증 상태를 어떻게 이어 붙이느냐, 이 한 문제가 세션과 JWT가 갈라지는 출발점이다.

## 세션: 서버가 기억한다

가장 자연스러운 해법은 서버가 기억하는 것이다. 로그인에 성공하면 서버가 메모리나 DB에 "세션 123번은 그 사용자, 로그인 상태"라고 적어 둔다. 그리고 사용자에게는 세션 ID `123`이라는 번호표만 쿠키로 준다. 이후 요청마다 브라우저가 이 쿠키를 자동으로 실어 보내고, 서버는 `123`을 받아 자기 저장소를 뒤져 누구인지 알아낸다.

핵심은 번호표에 아무 정보가 없다는 점이다. 진짜 정보는 전부 서버 안에 있고, 쿠키는 그것을 가리키는 표일 뿐이다. 이 번호표 구조와 세션 저장소 이야기는 [[cookie-and-session]]과 [[session-store]]에서 자세히 다룬다. 이 방식의 성질은 한 줄로 압축된다. 상태는 서버에 있다.

## 세션이 부담이 되는 지점

세션은 오래, 잘 작동했다. JWT는 세션이 틀려서 나온 것이 아니다. 환경이 바뀌면서 "서버가 기억한다"는 성질이 부담이 되는 상황이 생겨 나왔다. 배경을 네 가지 부담으로 보면 또렷하다.

첫째, 서버가 여러 대가 됐다. 사용자가 늘면 서버를 여러 대로 늘리는데, 세션 정보는 로그인을 처리한 그 서버 한 대의 메모리에만 있다. 다음 요청이 다른 서버로 가면 세션을 찾지 못한다. 이것을 풀려면 모든 서버가 세션을 공유해야 하고, 그러면 중앙 세션 저장소(보통 Redis)가 또 하나의 관리 대상이자 병목이 된다. 이 저장소 이야기는 [[session-store]]에서 다룬다.

둘째, 서버가 여러 종류가 됐다. 서비스를 작은 서버 여럿으로 쪼개면 요청 하나가 여러 서버를 거치고, 서버마다 사용자가 누구인지를 중앙 저장소에 매번 물어봐야 한다.

셋째, 브라우저 말고도 접속하는 쪽이 늘었다. 쿠키는 브라우저에서만 통하는 방식이다. 모바일 앱이나 다른 서비스가 API를 호출할 때는 쿠키 기반 세션이 잘 맞지 않고, 도메인이 다르면 쿠키를 주고받기도 까다롭다.

넷째, 제3자에게 권한을 안전하게 넘기는 문제가 커졌다. "구글 계정으로 로그인" 같은 흐름이 퍼지면서, 서버가 상태를 들고 있지 않아도 토큰 자체를 검사만 하면 신뢰할 수 있는 방식이 필요해졌다.

이 부담들의 공통 해답은 하나다. 서버에 물어보지 않아도 되도록, 정보를 토큰 안에 넣고 위조만 못 하게 만드는 것이다. 이것을 표준으로 정한 것이 JWT다. 정식 규격은 RFC 7519이고, JOSE라는 더 큰 규격 묶음(서명은 JWS, 암호화는 JWE, 키는 JWK)의 한 조각이다.

## 토큰이 스스로 증명하게 한다

세션은 상태를 서버에 두고 토큰은 그것을 가리키는 번호표였다. 이 발상을 뒤집는다. 상태를 토큰 안으로 옮긴다. 토큰이 "나는 이 사용자이고, 이 시각까지 유효하다"를 스스로 지니고, 서버는 그 토큰이 위조되지 않았다는 것만 확인한다. 서버는 아무것도 기억하지 않는다.

이 한 번의 뒤집기에서 JWT의 모든 장점과 모든 문제가 나온다. 장점은 서버가 저장소를 조회하지 않아도 된다는 것이고, 문제는 서버가 기억하지 않으니 한 번 내준 토큰을 도로 무효화하기 어렵다는 것이다. 둘 다 뒤에서 짚는다.

## JWT의 생김새

JWT는 점(`.`) 두 개로 나뉜 세 덩어리의 문자열이다.

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9        ← 헤더(Header)
.
eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyLCJleHAiOjE1MTYyNDI2MjIsImlzcyI6ImF1dGguZXhhbXBsZS5jb20iLCJhdWQiOiJhcGkuZXhhbXBsZS5jb20ifQ   ← 내용(Payload)
.
UDV_Z9h8RvDArR3awlQS5jhNedVt37InqigKqazwhaE ← 서명(Signature)
```

각 덩어리를 base64url로 풀면 아래처럼 디코딩된다.

헤더는 이 토큰이 무엇이고 어떻게 서명됐는지를 담는다.

```json
{ "alg": "HS256", "typ": "JWT" }
```

`alg`는 이 토큰을 어떤 방식으로 서명했는지다. 흔히 HS256(대칭)이나 RS256(비대칭)을 쓴다. 서명 방식과 키를 어디에 두고 어떻게 나누는지는 [[jwt-signing-and-keys]]에서 다룬다.

내용은 담고 싶은 정보 조각들이다. 이 조각 하나하나를 클레임(claim), 곧 "주장"이라 부른다.

```json
{
  "sub": "1234567890",
  "name": "John Doe",
  "iat": 1516239022,
  "exp": 1516242622,
  "iss": "auth.example.com",
  "aud": "api.example.com"
}
```

클레임은 두 종류다. `name`처럼 앱이 임의로 넣는 것은 표준이 아닌 커스텀 클레임이고, 나머지는 규격이 정한 표준 클레임이다. 표준 클레임은 몇 개만 알면 된다. `sub`(subject)는 이 토큰이 가리키는 사용자로 보통 사용자 id다. `iat`(issued at)는 발급 시각, `exp`(expiration)는 만료 시각, `iss`(issuer)는 발급 주체, `aud`(audience)는 이 토큰을 쓸 대상이다. 이 중 `exp`가 뒤에서 중요해진다.

서명은 위조 방지 도장이다. 헤더와 내용을 이어 붙인 문자열을 서버만 아는 비밀키로 해싱해서 만든다.

```
서명 = HS256( base64url(헤더) + "." + base64url(내용), 서버의 비밀키 )
```

여기서 초보자가 대부분 걸려 넘어지는 지점이 있다. 헤더와 내용은 암호화가 아니라 base64로 인코딩만 돼 있다. 누구나 풀어서 읽을 수 있다. jwt.io에 붙여넣으면 내용이 그대로 다 보인다. 그러니 JWT에 비밀번호나 민감정보를 넣으면 안 된다. 사실상 공개된 값이기 때문이다. 서명이 지키는 것은 비밀이 아니라 위조 불가다. 누구나 읽을 수 있지만 아무도 몰래 고칠 수는 없다. 서명은 무결성과 진위를 지키지, 내용의 비밀을 지키지 않는다.

## 서버가 토큰을 믿는 방법

여기가 세션과 가장 크게 갈라지는 대목이다. 세션이었다면 서버는 번호표를 받아 저장소를 뒤졌다. JWT는 뒤질 것이 없다. 대신 서버는 이렇게 한다.

받은 토큰에서 헤더와 내용을 떼어내고, 자기 비밀키로 서명을 그 자리에서 다시 계산한다. 다시 계산한 서명이 토큰에 붙어 온 서명과 같은지 비교한다. 같으면 발급 이후 아무도 내용에 손대지 않았다는 뜻이니 믿고, `exp`(만료)만 추가로 확인한다. 다르면 누군가 내용을 고쳤다는 뜻이니 거부한다.

왜 이것이 위조를 막는가. 공격자가 내용에 든 이름을 자기 것으로 바꾸면 서명도 달라져야 하는데, 올바른 서명을 새로 만들려면 서버의 비밀키가 필요하고 그것은 공격자에게 없다. 그래서 내용만 고치면 서명 검증에서 걸린다. 이 성질 덕분에 서버는 저장소를 조회하지 않고, DB에도 가지 않고, 계산 한 번으로 요청을 인증한다. 서버가 여러 대든 여러 종류든, 검증에 필요한 키(HS256이면 공유 비밀키, RS256이면 공개키)만 있으면 각자 알아서 검증한다. 앞의 네 가지 부담이 여기서 한꺼번에 풀린다.

실무에선 이 과정이 라이브러리 호출 한 번으로 끝난다.

```js
import jwt from 'jsonwebtoken'

try {
  // 넘기는 키만 방식에 따라 다르다: HS256=비밀키, RS256=공개키
  const payload = jwt.verify(token, key, { algorithms: ['HS256'] })
  // 여기 왔으면 서명 일치 + exp 유효. payload를 신뢰한다.
} catch (e) {
  // 서명 불일치나 만료면 throw 되어 여기로. 요청을 거부한다(401).
}
```

`verify` 한 번이 서명 재계산·비교·`exp` 확인을 다 한다. `algorithms`를 못 박는 것은 `alg`를 `none`으로 바꾸는 공격을 막기 위해서다. 서명 검증은 서버가 하고, 프런트는 토큰을 저장·전송만 한다.

## 세션과의 차이

차이는 딱 한 곳에서 갈린다. 상태가 어디 있느냐다. 세션은 상태가 서버에 있고 토큰은 그것을 가리키는 번호표라, 검증할 때 저장소를 조회한다. JWT는 상태가 토큰 안에 있고 서버는 위조만 검사하므로, 검증할 때 서명을 다시 계산할 뿐이다. 그래서 세션은 확장할 때 저장소를 공유해야 하고, JWT는 검증에 필요한 키만 있으면 된다. 대신 세션은 서버에서 지우면 즉시 무효가 되지만, JWT는 한 번 내주면 무효화하기 어렵다.

한 줄로 줄이면 이렇다. 세션은 서버가 기억하고, JWT는 토큰이 증명한다.

## 무상태가 만드는 대가

JWT는 만능이 아니다. 장점의 원천인 "서버가 기억하지 않는다"가 곧바로 세 가지 문제를 만든다. 셋 다 원인이 하나다.

첫째, 로그아웃과 강제 만료가 어렵다. 세션은 서버에서 지우면 그 즉시 무효다. 그런데 JWT는 서버가 아무것도 저장하지 않으니 한 번 발급한 토큰을 도로 무효화할 방법이 없다. 만료 시각 전까지는 계속 유효하다. 비밀번호를 바꿔도, 계정을 정지시켜도, 이미 나간 토큰은 살아 있다. 해결책은 하나같이 서버가 다시 무언가를 기억하게 만드는 방향이다. 만료를 아주 짧게 잡거나, 무효화할 토큰(보통 refresh token) 목록(denylist)을 서버에 두거나, 사용자별 토큰 버전을 두고 안 맞으면 거부하는 식이다. 뒤의 둘은 무상태라는 장점을 일부 반납하는 셈이다.

둘째, 그래서 refresh token이 필요하다. 첫째 문제가 그대로 refresh token의 존재 이유가 된다. 토큰은 탈취당해도 무효화할 수 없으니 수명을 짧게 해야 피해가 작다(access token, 예를 들어 15분). 그런데 15분마다 다시 로그인시키면 사용자가 견디지 못한다. 그래서 수명이 긴 refresh token을 따로 준다. 이것은 평소 요청에는 쓰지 않고 오직 새 access token을 받는 용도로만 쓴다. refresh token은 자주 쓰이지 않고 서버가 관리(저장·무효화)하므로, 이것으로 로그아웃과 강제 만료 문제를 어느 정도 되돌린다. 정리하면 access token은 자주 쓰지만 금방 죽고, refresh token은 가끔 쓰지만 오래 살며 서버가 통제한다. 짧은 수명이 주는 안전과 로그인 유지의 편의를 둘로 나눠 함께 잡는 방법이다. 다만 refresh token도 결국 하나의 토큰이라 탈취될 수 있고, 그 관리를 실제로 어떻게 하는지는 [[refresh-token-rotation]]에서 다룬다.

셋째, 토큰을 브라우저 어디에 둘 것인가. 정답이 없고 공격 유형을 맞바꾸는 선택이다. localStorage에 저장하면 자바스크립트로 쉽게 읽어 헤더에 실어 보낼 수 있어 편하지만, XSS(악성 스크립트 주입)에 뚫리면 스크립트가 토큰을 그대로 훔쳐 간다. httpOnly 쿠키에 저장하면 자바스크립트가 읽지 못하니 XSS로는 토큰을 빼 가지 못하지만, 쿠키는 자동 전송되므로 CSRF(사용자 몰래 요청 위조) 위험이 생겨 CSRF 방어를 따로 붙여야 한다. 무조건 안전한 곳은 없고, 어느 공격을 더 막을지를 고르는 문제다. 요즘은 localStorage를 피하는 쪽이 우세하다. XSS 한 번이면 토큰이 그대로 털리기 때문이다. 전역 상태 라이브러리의 persist도 대개 localStorage에 저장하므로 같은 위험을 진다. 흔한 절충안은 access token은 메모리에만 두고(새로고침 때 refresh로 다시 받는다), refresh token은 httpOnly·Secure·SameSite 쿠키에 두는 것이다. 쿠키를 쓰면 CSRF를 SameSite와 CSRF 토큰(double submit) 등으로 따로 막는다. access token을 httpOnly 쿠키에 두지 않는 것은, Authorization 헤더에 실으려면 JS가 토큰을 읽어야 하는데 httpOnly가 바로 그 읽기를 막기 때문이다.

## 요약

인증의 갈림은 상태를 어디에 두느냐 한 곳에서 갈린다. 세션은 상태를 서버에 두고 클라이언트에는 번호표만 주므로, 서버에서 지우면 즉시 무효가 되지만 서버가 상태를 들고 있어야 해 확장이 번거롭다. JWT는 상태를 서명된 토큰 안에 담아 서버가 위조만 검사하므로, 저장소 없이 확장하기 쉽지만 한 번 내준 토큰을 무효화하기 어렵다. 이 하나의 차이에서 장점도 문제도 모두 나오고, 실무는 짧은 access token과 서버가 추적하는 refresh token을 섞어 그 사이를 절충한다.
