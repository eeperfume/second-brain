# Control Plane (k8s의 두뇌)

k8s의 두뇌를 열어보자.

지금까지 YAML로 원하는 상태를 선언하면 마법처럼 Pod가 생기고 죽으면 살아나고 라우팅됐었다. 그렇다면 그 마법을 대체 누가 부렸을까? 상태는 어디에 저장됐을까? 이제 심층적으로 파고들어보자.

## 클러스터는 두 종류의 기계로 나뉜다

k8s 클러스터는 여러 대의 기계(노드, node) 묶음이다. 근데 노드가 두 종류로 나뉘는 것이다.

- Control Plane 노드 (두뇌): 무엇을, 어디에, 어떻게와 같은 결정을 내린다.
- Worker 노드 (일꾼): 실제로 Pod가 실행되는 곳이다.

참고로 관리형 k8s(EKS/GKE)를 쓰면 클라우드가 이 두뇌를 대신 운영해준다. 덕분에 Control Plane 노드인 두뇌를 안 봐도 되고, Worker 노드만 관리하면 되고 결제하면 된다. 앞서 말한 "운영 부담"을 클라우드에 떠넘기는 게 바로 이것이다.

## 두뇌(Control Plane) 안의 네 가지 부품

- API 서버 (kube-apiserver): 클러스터의 입구이다. 모든 게 여길 거쳐서 들어오고 나간다.
- etcd: 저장소이다. 모든 상태를 담은 key-value 자료구조의 DB이다. 여기서 모든 상태라 함은 원하는 상태와 현재 상태 모두를 뜻한다.
- 스케줄러 (kube-scheduler): 새 Pod를 어느 Worker 노드에 놓을지 결정한다.
- 컨트롤러 매니저 (kube-controller-manager): 제어 루프들이 사는 곳이다. Deployment 컨트롤러, ReplicaSet 컨트롤러 같은 것들이 전부 여기에 해당한다.

여기서 짚고 넘어가야 할 점이 있다.
- 갑자기 컨트롤러들이 등장했는데, 컨트롤러는 어디서 돌았을까? 사실은 컨트롤러 매니저 안에서 돌고 있었다. Ingress 컨트롤러만 여기에 살지 않는데, 밖에서 직접 세팅해 줘야 했던 게 그 예외였다.
- etcd가 뭘까? 두뇌의 기억 장치이다. 근데 중요한 규칙이 있다. etcd를 직접 만지는 건 오직 API 서버뿐이다. 다른 부품들은 etcd에 직접 안 가고 전부 API 서버를 통해서 읽고, 쓴다.

## 일꾼(Worker 노드) 안의 부품 세 가지

- kubelet: 각 Worker 노드의 손으로 생각하면 편하다. 내 노드에 무엇을 띄우라고 했는지를 API 서버(kube-apiserver)에서 보고, 실제로 컨테이너를 실행하거나 종료한다. 그리고 상태를 다시 보고한다.
- container runtime: 컨테이너를 진짜로 돌리는 엔진이다. 대표적으로 containerd가 있고, 다른 것들도 있다. 이 부품은 kubelet이 시킨다.
- kube-proxy: 그 노드에서 Service 라우팅을 담당한다. Service IP와 Pod IP 연결을 실제로 연결해준다.

## 전체 흐름

kubectl 명령어로 요청을 보내면 Control Plane(두뇌)로 가고, API 서버가 그 입구 역할을 한다. 그리고 API 서버는 etcd에 읽고 쓰기 때문에 연결돼 있다. etcd는 모든 상태를 기억한다. 즉 현재 상태나 원하는 상태 모두를 기억한다. 이제 Control Plane 안에 있는 다른 부품인 스케줄러(kube-scheduler)는 Pod를 어느 노드에 둘지 고민하고, 컨트롤러 매니저(kube-controller-manager)는 제어 루프들을 관리한다. 즉 집행자들을 관리하고 있다. 그런데 여기서 반드시 짚고 가야할 점은 스케줄러든 컨트롤러 매니저든 etcd에 직접 읽고 쓰는 게 아니라 다들 API 서버(kube-apiserver)만 쳐다보고 API 서버를 통해 읽고, API 서버를 통해 써주길 요청한다. 전부 API 서버를 통해서 N개의 Worker Node들이 생겨나고, 각 Worker Node들은 kubelet이라는 손과 container runtime라는 엔진을 갖게 된다.

## kubectl apply 하면 실제로 벌어지는 일

여기부터가 진짜 핵심이다. replicas: 3짜리 Deployment YAML을 kubectl apply 했다고 하자. 이제 두뇌(Control Plane) 안에서 벌어지는 연쇄를 따라가다 보면 지금까지 거쳐왔던 발자취가 전부 한 군데로 모인다.

1. kubectl apply 명령어를 통해 Deployment YAML을 API 서버(kube-apiserver)에 보낸다.
2. API 서버는 검증하고 etcd에 기록한다. 여기서 검증이라 함은 예를들어 Deployment my-web 3개를 v2버전으로 띄우길 원하는 상태를 검증하고, etcd라는 장부에 기록하는 것이다. 이후 저장을 마치면 "저장 완료"라고 응답한다.
3. Deployment 컨트롤러(kube-controller-manager 안)가 API 서버를 지켜보다가 새 Deployment를 발견하면 현재 상태와 유지해야 하는 상태를 비교해서 맞추려고 시도한다. 예를들어 새 이미지로 현재 상태를 맞춰야 한다고 가정했을 때 ReplicaSet이 없는 것을 인지하고 ReplicaSet 오브젝트를 만들 것이다. 그리고 API 서버를 통해 etcd에 기록해 달라고 요청할 것이다.
4. ReplicaSet 컨트롤러는 지켜보다가 "3개를 원하는데 Pod가 0개네?"하고 Pod 3개를 만들 것이다. 이 시점에는 아직 어느 노드에도 배정되지 않은 '대기중'상태이다.
5. 스케줄러는 '노드 없는 Pod'를 지켜보다가 각 노드의 남은 자원을 보고 "이 Pod는 노드2에서 띄우는 게 낫겠다" 판단하고, API 서버를 통해 etcd에 배정을 적는다.
6. WorkerNode로 와서 보면 이제 해당 노드의 kubelet이 "내 노드에 배정된 Pod가 있는지" 지켜보다가 발견하면 container runtime에게 이 이미지 받아서 컨테이너를 띄우라고 지시힌다. 그리고 상태를 API 서버에 보고한다.
7. 이제 Pod 3개가 진짜로 돌기 시작한다. 이 시점에는 현재 상태와 원하는 상태가 일치하게 된다. 끝이다.

그리고 self-healing도 같은 흐름이다. Pod가 하나 죽으면 kubelet이 죽었다고 API 서버에게 보고할 것이고, kube-controller-manager 안에 있는 ReplicaSet 컨트롤러는 API 서버를 바라보다가 컨트롤러가 "3개를 원하는데 2개가 됐네?" 발견하고, 새로운 Pod를 만들어 줄 것이다. 그리고 스케줄러를 통해 Pod를 어느 Worker Node에 배정할지 판단하고, 배정할 것이다. 그러면 다시 kubelet이 내 노드에 배정된 Pod가 있는지 지켜보다가 발견하게 되고 container runtime에게 실제로 컨테이너를 띄우라는 이 루프가 다시 돌게 된다.

## 서로 직접 명령하지 않는다 (이게 왜 튼튼한가)

여기서 반드시 알고 가야할 지점이 있다. 위의 흐름을 다시 봐라. 어떤 부품도 다른 부품에게 직접 명령하지 않는다.

- Deployment 컨트롤러가 스케줄러한테 "야 이거 배치해" 이렇게 직접 지시하지 않는다.
- 스케줄러도 Worker Node의 kubelet에게 "이거 띄워"라고 직접 지시하지 않는다.
- 다들 그냥 API 서버(kube-apiserver)를 통해 etcd의 공유 상태를 쳐다보고, 자기가 맡은 부분만 고쳐서 도로 적어둔다. 그럼 다음 부품이 그걸 보고 자기 일을 하는 것이다!

이렇게 하면 엄청 튼튼하게 쓸 수 있다. 얼만큼 튼튼하냐면 어떤 부품이 죽었다가 살아나도, 그냥 공유 상태를 다시 읽으면 자기가 뭘 해야 할지 바로 알게된다. 놓친 명령이나 사라진 메시지 같은 게 없다. 서로에게 "메시지"를 직접 보내지 않고, "현재 상태"만 보니까 스케줄러가 잠깐 죽어도, 살아나서 "노드 없는 Pod가 있네?"를 확인하고 마저 자기 일을 하면 그만이다. (참고로 이렇게 "지금 상태를 보고 반응"하는 방식을 level-triggered라고 부른다. 반대는 변화하는 순간의 이벤트에 반응하는 edge-triggered.)

트레이드오프로 보면 얻는 것은 다음과 같다. 압도적인 견고함(부품이 죽어도 자가 복구가 된다), 부품끼리 느슨하게 연결돼 있어서 단순하고, 교체하는 데 용이하다. 그리고 선언형으로 상태를 관리하고 싶었던 마음이 일관되게 유지된다.

잃는 것은 다음과 같다. 부품이 많고(두뇌만 4개이고, Worker Node마다 최소 2개에서 많게는 3개까지 갖게 된다), 마법이 여러 감시자에 흩어져 있어 디버깅하려면 이 흐름 전체를 알아야한다. 그리고 모든 게 루프이고, 즉각적이지 않아서 약간의 지연이 발생할 수도 있다. 그래서 관리형 k8s로 이 두뇌를 운영하는 걸 통째로 클라우드에 넘기는 선택지가 보편적으로 선택되는 것이다!
