# 네트워킹 (Service)

앞서 Pod는 소모품이라 죽고 새로 띄우는 과정에서 IP가 바뀐다는 문제가 있다고 했다. 그래서 프론트엔드는 대체 어느 IP로 백엔드 서버를 불러야 할지. 그리고 N개의 Pod 중에 어디에 요청을 보내야 할지 왜냐하면 트래픽 분산때문에라도 골고루 나눠 보내야 하는데 대체 어디로 요청을 보내야할지 알아야하는 문제가 있었다.

## 4. 네 번째 부품 Service

이 두 가지 문제를 한 번에 해결하는 부품이 바로 Service다.

Service를 쉽게 이해하자면 절대 안 변하는 대표번호 같은 것이다. 예를 들어 회사 대표번호를 떠올려보자. 대표번호로 전화를 걸면, 그때그때 자리에 있는 상담원 아무한테나 연결된다. 상담원이 퇴사하고 새로 들어와도(마치 Pod가 죽고 새로 떠도) 대표번호는 그대로고, 나는 누가 받을지 신경쓰지 않는다.

![[k8s-service.svg]]

Service는 이렇게 마치 대표번호같다. Service를 하나 만들면,
- 절대 변하지 않는 주소를 하나 주고, 고정된 가상 IP(ClusterIP)랑 고정 이름(DNS)을 준다. 뒤의 Pod가 아무리 죽고 살아도 이 주소는 그대로다.
- 그 뒤에 살아있는 Pod들한테 요청을 나눠 보내준다(로드 밸런싱). N개의 Pod 중 누구한테 갈진 Service가 알아서 판단한다.

그래서 프론트엔드는 그냥 `http://backend`처럼 이렇게 이름(DNS)으로 부르면 된다. 클러스터 안엔 DNS가 있어서 그 이름을 Service의 고정 IP로 바꿔주고, Service가 살아있는 Pod한테 연결해준다. 즉 "바뀌는 것(Pod)" 앞에 "안 바뀌는 문(Service)"을 하나 세운 것과 같다.

## 라벨과 셀렉터 — k8s의 숨은 접착제

그럼 Service는 뒤의 Pod를 어떻게 알아볼 수 있을까? 이게 바로 라벨이다. 이게 k8s의 숨은 접착제 같은 느낌을 준다!

여기엔 k8s 전체를 관통하는 핵심이 하나 있는데, 그게 바로 라벨(label)과 셀렉터(selector)의 연관관계이다.

```
Deployment와 Service가 각각 라벨을 어떻게 쓰는지 엮어보자. (오답노트)
- Deployment는 "라벨을 붙이는 쪽"이다. Pod를 찍어낼 때 template에 labels: app: my-web 도장을 찍어서 내보낸다. (동시에 자기가 관리할 Pod를 고르는 데도 셀렉터로 쓴다.)
- Service는 "라벨을 찾는 쪽/읽는 쪽"이다. selector: app: my-web으로 "그 도장 찍힌 Pod들"을 찾아서 요청을 보낸다.
즉 같은 app: my-web이라는 라벨 문자열이 둘 사이의 "약속"인 것이다. 하나는 붙이고, 하나는 읽고.

그리고 여기가 재밌는 부분인데, Service는 Deployment를 모른다! 서로 직접 아는 사이가 아니다. Service는 "Deployment야, 네 Pod 좀 줘"라고 하지 않는다. 그냥 라벨만 보고 맞는 Pod를 주워간다. 반대로 Deployment도 Service가 자길 보고 있는지 모른다. 둘은 서로를 모른 채, 라벨이라는 공통 약속 위에서만 만난다. 이게 "느슨한 결합"의 진짜 뜻이다. 서로를 지목하지 않으니, 한쪽을 갈아치워도 다른 쪽이 무너지지 않는다. (심지어 Service 여러 개가 같은 Pod들을 각자 라벨로 골라 가도 된다!)
```

k8s 부품들은 서로를 고정된 ID로 지목하지 않는다. 대신 "이런 라벨이 붙은 애들"이라고 느슨하게 가리킨다. 생각해보면 Deployment를 이해할 때 selector: app: my-web처럼 자기 Pod를 골랐었다! Service도 마찬가지로 똑같이 고른다.

```yaml
apiVersion: v1
kind: Service
metadata:
  name: backend    # <- 프론트가 부를 이름이다 (http://backend)
spec:
  selector:
    app: my-web    # * 이 라벨이 붙은 Pod들에게 요청을 보낸다.
  ports:
    - port: 80         # 이 Service의 80번 포트로 들어온 요청을
      targetPort: 8080 # Pod의 8080번 포트로 전달해라.
```

이 selector: app: my-web이 아까 Deployment가 찍어낸 Pod들의 라벨(app: my-web)이랑 같다! 그래서 Service는 그 라벨 붙은 Pod가 지금 몇 개든, IP가 뭐든 자동으로 추적할 수 있게 된다. Pod가 죽어서 목록에서 빠지고, 새로 떠서 목록에 들어오는 이 갱신 과정을 k8s가 실시간으로 해 준다. (내부적으로는 살아있는 Pod IP 목록을 Endpoints라는 데 계속 갱신한다)

```
Service는 뒤의 Pod가 죽고 새로 떠도 어떻게 안 놓치고 따라가는지 제대로 설명할 수 있도록 이 문단을 작성한다. (오답노트)
Service는 뒤의 Pod IP들을 한 번 적어놓고 쓰는 게 아니다. Service는 이 라벨 붙은 애들이 지금 누군지 끊임없이 다시 묻는다. 그래서 Pod가 죽어서 라벨 목록에서 빠지면 그 순간 라우팅 대상에서 빠지고, 새 Pod가 라벨 달고 뜨면 자동으로 그 라벨 목록에 들어온다. k8s가 살아있는 Pod IP 목록(Endpoints)을 실시간으로 갱신해주기 때문이다!
```

이게 왜 우아하냐면 Service는 특정 Pod를 "너!"하고 지목하지 않는다. "이 라벨인 애들"이라고만 해두니까, 뒤에서 Pod가 아무리 새로 뜨고, 죽는 과정을 통해 IP가 바뀌여도 연결이 안 끊기게 된다. 느슨하게 묶어놨기 때문에 밑에서 마음껏 죽고 살고해도 앱이 무너지지 않는 것이다. 라벨은 k8s 곳곳을 이어붙이는 접착제 같다. Deployment도 Service도 앞으로 나올 것들도 전부 다 이 라벨로 서로를 찾는다.

## Service의 종류

Service 종류를 가르는 기준은 "누가 이 문을 두드릴 수 있는가?"이다. Service에는 몇 가지 종류가 있는데, "안에서만 쓸 문이냐, 바깥에도 열 문이냐" 그 차이다.

| 종류 | 누가 접근 | 언제 쓰나 |
|---|---|---|
| ClusterIP (기본) | 클러스터 안에서만 | 내부 서비스끼리. 프론트 Pod → 백엔드 Service 같은 거. |
| NodePort | 서버(노드)의 IP:포트로 바깥에서도 | 급하게/단순하게 외부 노출. 투박함. |
| LoadBalancer | 인터넷 공개 IP | 클라우드에서 서비스를 진짜 세상에 여는 표준 방법. 클라우드가 로드밸런서(AWS ELB 등)를 붙여줌. |

대부분의 내부 통신은 ClusterIP로 충분하다. 외부에 열어야 할 때만 LoadBalancer를 쓴다.

여기서 트레이드오프가 하나 나온다. 외부에 열 서비스가 여러 개면? LoadBalancer는 하나당 클라우드 로드밸런서 하나라 돈이 하나씩 붙는다. 즉 서비스 10개 열려고 공개 IP를 10개 사는 건 낭비다. 게다가 보통은 이런 걸 원한다.

> mysite.com/api로 오면 백엔드 Service로, mysite.com으로 오면 프론트 Service로 보내줘. 주소(경로/호스트)를 보고 똑똑하게 알아서 나눠 보내줘.

이런 요구사항을 문 하나로 해 주는 부품이 바로 Ingress다. 이건 다음에 config/secret이랑 같이 다뤄보자. [[k8s-ingress-and-config]]로 이어진다.
