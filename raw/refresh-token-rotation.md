# refresh token 회전과 재사용 탐지

[[session-and-jwt]]에서 refresh token은 "서버가 관리(저장·무효화)한다"고만 해뒀다. 그 관리가 실제로 어떻게 도는지, 그리고 관리한다고 해서 탈취 자체를 막는 건 아니라는 문제를 여기서 다룬다.

## refresh token도 결국 하나의 토큰이다

access token을 짧게 두는 이유는 탈취돼도 금방 만료돼서 피해 보는 시간이 짧기 때문이었다. 그런데 그 대가로 만든 refresh token은 반대로 수명이 길다(며칠에서 몇 주). 오래 사는 토큰일수록 어디선가 새어 나갈 일도 그만큼 많아진다. localStorage에 있다면 XSS 한 번에 나가고, 네트워크 로그나 프록시 로그에 우연히 찍혀서 새는 경우도 있다.

문제는 여기서 생긴다. 훔친 refresh token은 진짜 사용자의 것과 구별이 안 된다. [[session-store]]에서 짚었듯, 이런 류의 토큰은 가진 사람을 곧 주인으로 취급하는 bearer token이라, "이 토큰을 들고 온 게 진짜 그 사용자인지 도둑인지"는 토큰 자체만 봐서는 알 길이 없다. denylist에 올리는 건 이미 도난을 안 뒤의 대응이지, 도난을 알아채는 방법은 아니다. 그럼 서버는 애초에 도난을 어떻게 눈치채는가.

## 한 번 쓰고 버리는 표

해법은 refresh token을 여러 번 못 쓰게 만드는 것이다. access token을 새로 받으려고 refresh token을 서버에 내면, 서버는 새 access token과 **새 refresh token**을 함께 내주면서 방금 받은 옛 refresh token을 그 자리에서 무효화한다. 클라이언트는 새 refresh token으로 갈아 끼우고, 다음 갱신 때는 이것을 쓴다.

이렇게 하면 refresh token은 지하철 일회용 승차권과 비슷해진다. 개찰구를 한 번 통과하면 그 표는 죽고, 다음 갱신은 새 표로 한다. 정상적인 사용자라면 매번 최신 표로 자연스럽게 갈아타므로 아무 문제가 없다.

## 재사용이 곧 도난 신호다

이렇게 돌리면 쓸모 있는 성질이 하나 생긴다. 정상적인 흐름에서는 이미 쓴(옛) refresh token이 다시 제출될 일이 없다. 진짜 사용자는 회전할 때마다 최신 토큰으로만 계속 갈아타지, 죽은 옛 표를 다시 꺼내 쓸 이유가 없기 때문이다.

그런데 이미 무효화된 refresh token이 서버에 다시 들어온다면 둘 중 하나다. 오래 잠들어 있던 탭이 옛 토큰으로 갱신을 시도했거나(뒤에서 볼 동시 요청 문제가 대표적이다), 공격자가 예전에 훔쳐 둔 refresh token을 지금 꺼내 쓴 것이다. 서버는 이 둘을 구별할 수 없으니 나쁜 쪽으로 가정한다. 정상 사용자는 이미 다음 토큰으로 넘어가 있는데 죽은 토큰이 또 나타났다는 건, 같은 토큰이 서로 다른 두 곳에서 쓰였다는 뜻이고 그중 하나가 도둑일 수 있기 때문이다.

그래서 재사용이 감지되면 서버는 "혹시 모르니 한 번 더 봐주자"가 아니라, 즉시 **그 refresh token이 속한 사슬 전체를 무효화**한다. 지금 요청 하나만 막는 게 아니라, 그 로그인에서 나온 모든 후속 토큰까지 통째로 죽인다. 진짜 사용자도 이 순간 로그아웃되지만, 다시 로그인하면 그만이다. 도둑은 여기서 끝이다. 재로그인 한 번의 불편으로 도난 전체를 끊는 셈이라 남는 장사다.

## 토큰 패밀리: 사슬을 하나로 묶기

이 무효화가 "그 refresh token 하나"가 아니라 "그 사슬 전체"에 걸리려면, 회전으로 이어지는 토큰들이 서로 같은 사슬임을 알 수 있어야 한다. 그래서 로그인해서 첫 refresh token을 낼 때 `family_id`(또는 session_id) 하나를 함께 발급하고, 회전할 때마다 새로 나오는 토큰도 같은 `family_id`를 물려받는다. 이렇게 한 로그인에서 나온 토큰 무리가 token family다.

서버가 저장소에 갖고 있는 레코드는 대략 이런 모양이다.

```
token_id: t3
family_id: fam-8f2a
user_id: 1067
status: active        # active | rotated | revoked
issued_at: ...
```

토큰을 회전할 때 `t3`의 `status`를 `rotated`로 바꾸고 `t4`를 `active`로 새로 넣는다. 나중에 `status: rotated`인 `t3`가 다시 들어오면, `t3`의 `family_id`로 그 사슬에 속한 레코드를 전부 찾아 `revoked`로 바꾼다. 개별 토큰이 아니라 `family_id`를 기준으로 지우는 것이 핵심이다. 이렇게 저장소에 상태를 두는 것 자체가, [[session-and-jwt]]에서 말한 "무상태라는 장점을 일부 반납한다"의 실제 모습이다. refresh token 쪽은 애초에 완전한 무상태로 굴리지 않는다.

## 그래서 refresh token은 흔히 JWT가 아니다

여기서 자연스럽게 갈리는 선택이 하나 있다. access token은 서명만 검증하면 끝나니 JWT가 잘 맞지만, refresh token은 매번 서버 저장소를 들여다봐야(살아 있는지, 재사용인지) 하므로 "저장소를 안 봐도 된다"는 JWT의 장점이 애초에 성립하지 않는다. 그래서 실무에서는 refresh token을 JWT로 만들지 않고, 그냥 추측 불가능한 랜덤 문자열(opaque token)로 발급해 DB나 Redis에 `token → family_id, status, user_id` 매핑만 저장하는 경우가 흔하다. 서명도 클레임도 필요 없다. access token은 서명된 JWT로 무상태 검증을 누리고, refresh token은 어차피 상태를 쥘 수밖에 없으니 차라리 단순한 불투명 토큰으로 두는 것이다. 세션 ID와 성격이 다시 비슷해진다.

## 회전이 만드는 새 문제: 동시 요청

회전 덕에 재사용을 탐지할 수 있게 됐지만, 대신 조심할 것이 하나 생긴다. refresh token이 한 번만 유효하므로, 같은 refresh token으로 요청 두 개가 거의 동시에 들어오면 하나만 성공하고 나머지는 이미 쓴 토큰을 낸 셈이 되어 재사용으로 오인될 수 있다. 브라우저 탭 여러 개가 동시에 만료를 감지해 각자 refresh를 시도하는 상황에서 실제로 이렇게 된다. 이 경합을 다루는 얘기(refresh 요청을 하나로 묶기, 짧은 유예 시간을 두기 등)는 별도로 파고들 만하다.

## 요약

refresh token은 access token보다 오래 사는 만큼 탈취될 위험도 그만큼 오래 간다. 그 위험에 대응하는 방법이 회전이다. 쓸 때마다 새 토큰으로 바꾸고 옛 토큰을 죽이면, 이미 죽은 토큰이 다시 나타나는 것 자체가 도난 신호가 된다. 이 신호를 잡으면 토큰 하나가 아니라 그 로그인에서 나온 token family 전체를 무효화해 진짜 사용자와 도둑을 함께 끊는다. 이 상태 관리 때문에 refresh token은 흔히 JWT가 아니라 서버가 직접 추적하는 불투명 토큰으로 구현된다.
