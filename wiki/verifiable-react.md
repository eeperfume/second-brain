# 검증 가능한 React 컴포넌트

> 한 줄: 컴포넌트가 자기 상태를 렌더된 DOM(표면)에 `data-verify-*`로 드러내, 테스트 러너·사람·AI 에이전트가 **같은 한 면을 읽고 같은 판정**에 이르게 하자는 발상.

라이브러리가 아니라 발상이다. 코딩 에이전트에게 한 번 건네 읽히는 글에서 증류했다.

## 문제

- 단위 테스트는 내부 상태·props를 들여다봐서, 동작이 그대로여도 리팩터하면 깨진다.
- 스냅샷 테스트는 마크업을 통째로 얼려서, 무해한 수정에도 깨진다.
- 수동 클릭은 한 경로를 한 번 확인할 뿐이고, "괜찮아 보임"은 아무것도 증명하지 않는다.
- 넷 다 구현을 들여다볼 뿐 관찰 가능한 출력을 안 본다. 그래서 세 판정자(테스트 러너·사람·에이전트)의 판정이 서로 어긋난다.

## 표면(surface)과 data-verify-*

컴포넌트가 렌더 결과로 겉에 드러낸 면을 **표면**이라 부른다. 상태를 `data-verify-*` 속성으로 DOM에 문자열로 노출하고, 모두가 그 한 면을 읽는다.

- 값은 문자열로 비교하므로 가격은 부동소수 대신 정수 센트로 적는다(예: `4997`).
- 아무것도 안 드러내는 조용한 컴포넌트는 성공이 아니라 실패로 친다.

```html
<section
  data-verify-unit="CartSummary"
  data-verify-item-count="3"
  data-verify-subtotal-cents="4997"
  data-verify-empty="false"
>
  ...
</section>
```

## 엔진과 명세

- **엔진**: 재사용 기계. 컴포넌트를 정해진 상태로 마운트하고, 상호작용을 일으키고, 표면을 읽고, 검사해서, 판정 하나를 낸다. 코드가 아니라 규칙으로 정의된다. 횡단 검사(속성 존재·접근성·성능 예산: 번들 200KB 이하, 첫 화면 2.5초 이내, 요청 50개 이하)는 엔진에 둔다.
- **명세**: 컴포넌트마다 쓰는 유일한 것. 상태 몇 개를 이름 짓고, 무엇이 각 상태를 만들며, 무엇이 참이어야 하는지 단언한다. 올바른 것과 버그가 실제로 갈리는 상태를 골라라. 깨질 수 있는 것마다 계기를 달아라.

## 판정 4종

- **PASS** / **FAIL**
- **BLOCKED**: 볼 수조차 없었다(마운트 안 됨·설정이 예외를 던짐·페이지 미완성). FAIL과 섞으면 안 된다.
- **SKIP**: 검사할 게 없다.

바탕 편향은 **확신이 안 서면 실패로 친다**이다. 거짓 통과는 비싸고 거짓 실패는 싸다. 관찰하지 못한 것은 PASS가 아니라 BLOCKED로 돌아온다.

## 거짓을 잡는지 증명한다

항상 초록인 스위트는 아무것도 증명하지 않는다. 두 규칙:

1. 모든 컴포넌트에 해피패스 밖 케이스가 최소 하나.
2. 어딘가 **일부러 실패하도록 만든** 케이스가 최소 하나, 그리고 스위트가 그게 실패함을 단언한다. 이건 검사기 자신을 검사한다. 검사가 느슨해지면 이 케이스가 시끄럽게 깨진다.

할인 반올림 버그 예: 1000센트는 반올림·절삭이 모두 900이라 버그가 숨는다. 999센트는 99.9로 갈려, 올바른 코드는 반올림해 100(소계 899), 버그는 잘라내 99(소계 900)가 된다. 1센트짜리 분기가 버그를 드러낸다.

## 판정은 한 번만 계산한다

한 번 계산해 기록한다. 대시보드·CI 게이트·에이전트 핸들은 모두 그 하나의 저장된 결과를 보는 뷰다. "CI는 초록인데 에이전트는 빨강"을 막는다.

- 에이전트가 코드를 실행하지 않고도 답을 읽게 둔다(브라우저에서 돌면 DOM 노드, 아니면 결과 파일).
- 기록에는 행동에 필요한 정보를 담는다: 어느 단위인지, 무엇을 기대했는지, 표면이 무엇을 보고했는지.

## 동작(is-right)과 겉보기(looks-right)는 다르다

- **is-right**: 작동하는가. 기계의 몫, 헤드리스로 판정을 낸다.
- **looks-right**: 사람 눈·스크린샷, 진짜 픽셀이 있는 곳에서만.
- looks-right가 is-right를 대신하게 두지 마라.

## 스택에 맞추기

- 표면은 결국 브라우저에 도달하는 무엇이든이다. Next.js(서버 렌더 컴포넌트)·Remix(loader·action)도 마찬가지다.
- 브라우저에 안 닿는 서버 전용 로직은 순수 함수로 빼서 직접 검증하거나, 미검증으로 표시한다.
- 타이밍: 상호작용을 일으키고, 안정될 때까지 기다린 뒤, 읽는다.
- 가상 스크롤(윈도잉) 목록은 화면에 보이는 행을 세지 말고 상태의 총계를 보고한다.

## 덧붙임

일부러 추상적으로 남긴 글이다. 정확한 속성 이름·러너·마운트 방식은 프로젝트마다 다르다. 나머지는 에이전트가 이어 간다.
